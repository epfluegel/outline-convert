  - Confidentiality #slide #CI7100 #CI7160-intro #CI7130_Intro #basic
    - Definition: computer related assets must be accessed only by authorised parties
    - Synonyms: secrecy, privacy
    - Example: you need to prevent hackers from accessing personal data in your social network account
    - There may be different levels of confidentiality for different users
  - Integrity #slide #CI7100 #CI7160-intro #CI7130_Intro #basic
    - Data Integrity:
      - Data has not been modified by unauthorised parties
      - Example: file transfer, email, computer viruses
    - Origin Integrity (Authentication):
      - We have proof about origin of a message
      - Example: IP address of the machine the e-mail was sent from
    - Non-repudiation:
      - Sender (or recipient) of a message cannot deny having been involved in message transfer
      - This is more complex (usually involves 3 parties)
  - Availability #slide #CI7100 #CI7160-intro #CI7130_Intro #basic
    - Definition: assets are accessible to authorised parties at appropriate times
    - In other words: legitimate access should always be possible
    - Example: a website should be available 24/7.
    - Different availability levels

    - Hardware Vulnerabilities #slide #CI7130_Intro
      - Physical Vulnerabilities
        - Hardware is composed of physical objects -- it is simple to add, change, destroy or remove parts.
        - This means that computer devices might be prone to a physical attack.
        - Computers have been stolen, kicked, punched, burnt, frozen, shot at.. the https://youtu.be/algWjS2yiyE) !
        - Before the University tightened its physical controls, every semester at least one projector used to be stolen!
      - Microprocessor Design Flaws
        - Recent huge issue: flaws in Intel/ARM chip design ("Meltdown" and "Spectre")
        - This is actually a combination of hardware and software vulnerabilities
    - Data Vulnerabilities #slide #CI7130_Intro
      - Enormous amounts of data around (big data), connected and exposed to the Internet, and stored on corporate clouds.
      - Data can be human-readable (text) or in binary format.
      - Need to think of data at rest, and in transit (computation/memory, hard disk, network, brain).
      - Data attacks are more widespread than software attacks.
        - Unprotected data is "easy" to read and intrinsically vulnerable to all threat outcomes (interception, interruption, modification).
        - Furthermore, it can be easily duplicated.
    - Software Vulnerabilities #slide #CI7130_Intro
      - Software is also data, so all data vulnerabilities apply
      - Weaknesses can appear during different stages of software development and deployment life cycle:
        - Source Code
          - Coding Flaws: mistakes in the design of source code, weak cryptographic implementations (ignoring standards).
        - Compiled (Binary) Code
          - Inherently vulnerable to malicious exploits (buffer overflow, SQL injection).
          - Expanded attack surface due to frequent use of common libraries.
        - Deployed Application
          - Misconfiguration: weaknesses such as security patches out of date, weak security settings (e.g., default passwords).
      - Software vulnerabilities are indexed in the CVE catalogue on cve.org.
    - People Vulnerabilities #slide #CI7130_Intro
      - People are often the weakest link.
      - Typical exploit: phishing/spam emails.
      - Effectiveness is often underestimated.
      - This is particularly problematic if the targeted people are key individuals within an IT system, such as system administrators or lecturers.
      - Vulnerabilities:
        - Lack of security awareness -- people tend to use computers irresponsibly.
        - Lack of skills, training, and expertise -- not having the technical know-how in security.
        - Personality Weaknesses
          - Laziness and forgetfulness -- writing down or reusing passwords, forgetting to make backups, etc.
          - Naivety -- if this persists despite training, people are an ideal target for social engineering.

  - THREATS AND ATTACKS #wfe-ignore-item
    - More About Threats #slide #CI7130_Intro
      - Threat versus Attack
        - An attack is the implementation of a event (= threat event)
      - Threat Actors
        - Ultimately, security attacks are carried out by real people
        - They have a motive
      - Threat Landscape
      - Threat Profiling
    - Threat Outcomes #slide #CI7130_Intro
      - There are four outcomes, threatening CIA triad:
        - Interception (synonyms: disclosure, evesdropping, snooping) -- unauthorised access to asset
        - Interruption -- an asset becomes unavailable
        - Modification -- tampering with an asset
        - Fabrication -- creation of new objects in a computer system
      - Examples:
        - copying data, stealing a password
        - destroying hardware device, corrupting a file
        - changing values in a database
        - adding records to a database, creating a trap door
    - Threats to Data  #slide #CI7130_Intro
      - Disclosure (Leaking):
        - Doxing -- Stealing data and dumping it on the Internet, such as on PasteBin
        - Credential Stuffing -- get hold of login information, and try it on different accounts
      - Malicious Modification:
        - Deletion -- accidental or deliberate
        - Salami attack -- modify only small amounts of data at a time, but then do this very often, examples:
          - Credit card theft
          - Transferring rounding errors in bank transactions to own account (see movie "Office Space")
        - Ransomware -- encrypt the victim's data, or threaten to publish it, unless a ransom is paid
      - Loss of Ownership
    - Threats and Software #slide #CI7130_Intro
      - Threats to Software
        - Source Code Disclosure: source code can leak, be stolen or reverse engineered
        - Exposure to Virus Infection: executable (binary) code can be modified by a virus
      - Software as Threat
        - Malware: Software that may itself have hidden or open malicious or undesirable side effects, or contain back doors
        - Terminology: Zero-day exploit -- the first attack of a newly found vulnerability (see: Google Project 0)
    - Cyber Threat Actors (Attackers) #slide #CI7130_Intro #new #wfe-ignore-outline
      - White Hat (Ethical) Hackers -- non-malicious attackers employed to find vulnerabilities in an organisation to improve security.
      - Black Hat Hackers (Computer Criminals) -- malicious attackers who want to profit financially.
      - Grey Hat Hackers -- somewhere between White Hat and Black Hat. For example, they might carry out ethical hacking activities without consent.
      - Script Kiddies -- non-malicious but thoughtless and haphazard (teenage) users. Find and run scripted security attacks to be entertained or thrilled.
      - Researchers (Cryptanalysts) -- non-malicious attackers who focus on finding security loopholes offline.
      - Other terms often used: #wfe-ignore-outline
        - Hacktivists: Attack for political or social causes.
        - Cybercriminals: Attack for financial gain.
        - Nation-states: Attack for political, military or economic espionage purposes.
        - Insiders: Attack for revenge, financial gain, or ideology.
        - Script kiddies: Attack for fun or to impress others.
        - Terrorists: Attack for ideological or political reasons.
        - Competitors: Attack for business advantage.
        - Hackers, crackers, malicious insiders, industrial spies, cybercriminals, hacktivists, and cyber-terrorists.

    - Risk #slide #CI7130_Intro
      - Risk occurs if both a threat and a matching vulnerability exist.
      - Risk arises if there is a probability of damage to our assets, arising from an attack
      - Realistic purpose of computer security: minimise risks (rather than achieving complete security)
      - Risk can be modelled, using appropriate risk analysis frameworks, as a function of:
        - (i) the adverse impacts that would arise if the circumstance or event occurs;
        - (ii) the likelihood of occurrence.

  - CONTROLS #wfe-ignore-item
    - More About Controls #slide #CI7130_Intro #basic
      - A control is an action, procedure, device or a technique used for protection. #control
      - We can discuss controls by considering the following aspects:
        - Design principle -- principle of the mitigation technique
        - Implementation -- how the control is realised
        - Strength -- how successful they might protect against attacks
        - Effectiveness -- to what extent they reach their full potential strength
        - Strategy -- how they are strategically designed, in a hostile environment
    - Controls -- Design principle #slide #CI7130_Intro #basic
      - Preventative or proactive control:
        - Preempt -- by blocking the attack, neutralising the threat, closing the vulnerability, or all of it
        - Deter -- by making the attack harder
        - Deflect -- by making another target more attractive
      - Detective control:
        - Detect the attack -- either as it happens or sometime afterwards
      - Reactive control:
        - Corrective or retrospective control: recover from the effect of an attack
    - Controls -- Implementation #slide #CI7130_Intro
      - Physical (Traditional) Controls -- e.g. guards, locks, backup copies
      - Hardware Controls -- e.g. certain types of firewalls, smartcards, hardware authenticators
      - Software Controls -- e.g. encryption, protection in operating systems, protocols, virus scanners, security patches
      - Administrative Controls -- e.g. security policies or security assessment frameworks used in an organisation, training, nudging
    - Controls -- Strength #slide #CI7130_Intro
      - Principle of Overlapping Control (Layered Defense)
        - Controls can be strengthened by using overlapping controls
        - Example: in a computer lab, use swipe cards and locks attached to projectors
      - Principle of Adequate Protection
        - Assets relate to cost -- strenght of protection should be consistent with the asset value
        - They might lose their value after some time -- only protect until loss of value
    - Controls -- Effectiveness #slide #CI7130_Intro
      - Principle of Effectiveness
        - No control is 100% safe -- quality of control will have impact on losses due to attacks
        - Controls are only effective, if they are used
        - Many users are unaware of need for security/unwilling to cooperate
        - Notion of "security theatre"
      - Examples
        - Detection rate of IDS
        - Comprehensiveness of firewall rules
        - Quality of software update policy
        - Who ever changes his/her passwords?
        - Are the swipe cards for our computer labs effective?
    - Controls -- Strategy #slide #CI7130_Intro
      - Principle of Weakest Link
        - A control can be no stronger than its weakest component
        - Example: we set up an elaborate firewall for the network, but forget to protect a wireless access point
      - Principle of Easiest Attack
        - The attacker will find the for him easiest loophole, not necessarily the most obvious one!
        - Example: spear-phishing
      - Principle of Periodic Review
        - Periodic review and re-evaluation is necessary to ensure the sustained effectiveness of a control
        - Example: re-imaging lab computers during term break
    - Should A Control Be A Worry-Eater? #slide #CI7130_Intro

                - Types of Attacks on Encryption Schemes
                  - Ciphertext-only Attack (COA)
                    - Adversary only has access to ciphertexts.
                    - Goal: Infer plaintext or key from ciphertext.
                  - Known Plaintext Attack (KPA)
                    - Adversary has some pairs of plaintext and corresponding ciphertext.
                    - Leverages known mappings to find the key or decrypt other ciphertexts.
                  - Chosen Plaintext Attack (CPA)
                    - Adversary can choose plaintexts and obtain corresponding ciphertexts.
                    - Used to learn about the encryption algorithm or key.
                  - Chosen Ciphertext Attack (CCA)
                    - Adversary can choose ciphertexts and see corresponding plaintexts.
                    - Helps in learning weaknesses in decryption to retrieve the key.
                - Advanced Attacks on Encryption Schemes
                  - Adaptive Chosen Plaintext Attack (Adaptive CPA)
                    - Adversary chooses new plaintexts based on prior responses.
                    - Aims to adaptively refine knowledge of the key or encryption method.
                  - Adaptive Chosen Ciphertext Attack (Adaptive CCA)
                    - Adversary chooses ciphertexts adaptively based on previous decrypted responses.
                    - Commonly used to exploit decryption oracle weaknesses.
                  - Side-Channel Attack
                    - Exploits physical information like timing or power consumption.
                    - Often bypasses mathematical defenses by targeting hardware.
                  - Brute Force Attack
                    - Tries all possible keys until the correct one is found.
                    - Generally impractical for strong encryption due to high computational cost.

        - Strength/Effectiveness of a Control
          - Motivation
            - No control is 100% safe
            - Quality of control will have impact on losses due to attacks
          - We define the strength of a control \(\gamma\) as the quotient of the recovered server value \(R_i\) and the damage level \(L_i\): \[ l(\gamma)_i = R_i / L_i \in [0,1], \]
          - Examples
            - Detection rate of IDS
            - Comprehensiveness of firewall rules
            - Quality of software update policy
        - Controls -- Strenght #CI7130_Intro
          - Principle of Overlapping Control (Layered Defense)
            - Controls can be strengthened by using overlapping controls
            - Example: in a computer lab, use swipe cards and locks attached to projectors
          - Principle of Adequate Protection
            - Assets relate to cost -- strenght of protection should be consistent with the asset value
            - They might lose their value after some time -- only protect until loss of value

    - The basic methodology in computer security is to first think about potential weaknesses in a given system.
      - In order to better understand these security weaknesses, we have to define what we want to protect -- this is called an asset.
      - Assets have fundamental security requirements.
        - Confidentiality -- only authorised parties have access to our assets.
        - Integrity -- assets have not been modified by unauthorised parties.
        - Availability -- assets are available to authorised parties at appropriate times.
        - Illustration: security is CIA
          - ![](https://www.dropbox.com/s/i12op0g3xmcby27/CIA.png?dl=1)
      - A vulnerability is a potentially exploitable weakness ("loophole") of an asset.
      - A threat is a set of circumstances with potential to exploit a vulnerability and create harm to assets.
    - This is followed by the design of suitable controls (countermeasures) in order to mitigate attacks.
      - An attack is the implementation of a threat.
      - A control is an action, procedure, device or a technique used for protection.
    - It is important to note however that it is impossible to guarantee absolute security -- we can only hope to reduce risk.
